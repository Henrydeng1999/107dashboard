# 04 四人团队协作方案

## 目标

本项目是比赛原型，团队协作的首要目标是快速形成一个稳定、可演示、能讲清楚价值的 Dashboard。四个人应围绕同一个可运行版本协作，避免每个人分别完成孤立模块，最后无法集成。

比赛阶段的产品闭环：

```text
填写作业参数
  -> 提交 Slurm 作业
  -> 查看排队和运行状态
  -> 查看 stdout/stderr 日志
  -> 取消或克隆作业
  -> 查看基础资源统计
```

## 协作平台

建议使用少量工具，各自承担明确职责：

| 平台 | 用途 | 不用于 |
| --- | --- | --- |
| Gitee 仓库 | 代码、文档、版本历史 | 即时聊天 |
| Gitee Issues | 功能任务、Bug、验收标准 | 临时口头安排 |
| Gitee Pull Requests | 代码评审、合并和变更说明 | 长篇产品讨论 |
| 微信或 QQ 群 | 快速沟通、紧急问题、会议通知 | 保存最终技术结论 |
| 腾讯文档或飞书文档 | 比赛材料、演示稿、会议记录 | 替代代码仓库文档 |
| Figma、即时设计或纸面草图 | 页面流程和界面原型 | 保存后端接口定义 |

重要结论最终要回写到 Gitee 的 README、`docs/`、Issue 或代码中。群聊记录不能作为唯一项目资料。

## 四人分工

四个人可以设置主要负责人，但所有人都应理解完整演示流程，并能处理自己模块与其他模块的接口。

### 成员 A：产品与前端

- 负责页面结构、交互流程和视觉一致性；
- 实现作业列表、提交表单、详情页、日志页和资源统计视图；
- 与后端共同维护 API 数据结构；
- 整理比赛演示流程和用户故事。

主要交付：

```text
frontend/
界面原型
演示流程
前端接口适配
```

### 成员 B：后端与数据

- 搭建 FastAPI 服务；
- 设计作业、状态快照和操作记录的数据模型；
- 提供提交、列表、详情、日志、取消和克隆 API；
- 使用 SQLite 保存比赛原型所需元数据；
- 统一错误响应和日志。

主要交付：

```text
backend/api/
backend/models/
backend/services/
数据库迁移或初始化逻辑
```

### 成员 C：Slurm 集成

- 封装 `sbatch`、`squeue`、`sacct` 和 `scancel`；
- 生成受控的作业脚本；
- 解析 Job ID、状态、退出码、资源和排队原因；
- 负责日志路径和作业目录约定；
- 在算力平台验证 QoS、分区和 GPU 数据。

主要交付：

```text
backend/slurm/
scripts/
Slurm mock 数据
平台验证记录
```

### 成员 D：集成、测试与比赛材料

- 维护本地和算力平台启动方式；
- 编写关键接口和流程测试；
- 负责前后端联调、演示数据和故障兜底；
- 维护 README、环境记录和比赛说明；
- 准备演示视频、答辩材料和备份方案。

主要交付：

```text
tests/
deploy/
docs/
演示数据和演示脚本
```

## 交叉负责原则

主要负责人不等于唯一负责人。建议建立以下搭档：

| 工作 | 主负责人 | 评审人 |
| --- | --- | --- |
| 页面与交互 | A | D |
| 后端 API | B | A |
| Slurm 集成 | C | B |
| 部署与测试 | D | C |

每个关键模块至少有两个人理解。比赛演示不能依赖某一个人临时在线修复。

## Gitee 工作流

### Issue

所有超过半小时的工作都建立 Issue。Issue 至少写清楚：

```text
目标
用户能看到的结果
实现范围
验收标准
负责人
截止时间
```

推荐标签：

```text
frontend
backend
slurm
test
docs
bug
demo
priority-high
```

### 分支

`master` 始终保持可演示。功能从短期分支开发：

```text
feature/job-list
feature/job-submit
feature/slurm-adapter
fix/log-tail
docs/demo-script
```

不要四个人长期维护四个大分支。分支应尽快通过 Pull Request 合并，减少最后集中解决冲突的风险。

### 提交

提交信息使用简单统一格式：

```text
feat: add job submission form
fix: handle missing Slurm log file
docs: update demo workflow
test: cover job status mapping
chore: update dependencies
```

一次提交尽量只包含一个主题，不提交密码、Token、SSH 私钥、真实用户日志或生产配置。

### Pull Request

除紧急文档修正外，功能代码通过 Pull Request 合并。PR 说明应包含：

```text
解决什么问题
主要修改
如何验证
界面截图或接口示例
已知限制
关联 Issue
```

至少一名队友检查后合并。比赛临近时可以缩短评审流程，但不能跳过基本运行验证。

## 通知方式

Gitee 没有可靠的项目成员任意群发邮件功能。建议组合使用：

- Gitee Issue 保存正式公告和任务；
- Issue 或 PR 中 `@用户名` 通知负责人；
- 微信或 QQ 群发布紧急提醒；
- 每次阶段性决定在群里通知，并把最终结果回写到仓库；
- 重要节点固定一条公告 Issue，记录版本、演示地址和待办事项。

不要只依赖 Gitee 邮件，因为成员可能没有开启对应邮件通知。

## 每日协作节奏

比赛开发阶段建议每天一次十分钟同步：

```text
昨天完成了什么
今天准备做什么
当前阻塞是什么
是否影响其他成员接口
今天结束前可演示什么
```

同步后更新 Issue 状态。每天至少产生一个可运行版本，避免连续多天只有局部代码、没有完整演示。

## 集成节奏

推荐按纵向功能闭环集成，而不是先分别做完整前端和完整后端：

1. 静态作业列表和后端健康检查；
2. 使用 mock 数据打通作业列表；
3. 接入真实 `squeue` 和 `sacct`；
4. 打通最小作业提交；
5. 增加日志查看和取消；
6. 增加克隆和资源统计；
7. 冻结功能，集中修复演示问题。

每一步都应形成可以打开页面、点击操作并看到结果的版本。

## AI 辅助开发

比赛鼓励使用 AI 协助编写项目。团队可以用 AI：

- 整理需求和拆分 Issue；
- 生成代码骨架和测试；
- 解释 Slurm 输出和错误；
- 检查接口一致性；
- 维护 README 和比赛材料；
- 辅助代码评审和问题排查。

AI 生成内容必须由负责人验证。不能把未检查的命令直接用于删除文件、取消作业、修改权限或提交生产任务。AI 辅助开发不改变成员对自己提交代码的责任。

## 比赛演示保障

比赛前准备三套路径：

```text
A：连接真实算力平台并现场提交作业
B：读取预先提交的真实作业和日志
C：使用脱敏 mock 数据完成离线演示
```

同时保存：

- 一个成功作业；
- 一个排队作业示例；
- 一个失败作业及日志；
- 一份资源统计样例；
- 一段完整演示视频；
- 当前可演示版本的 Git tag 或 commit。

现场网络、队列和 GPU 状态不可控，演示方案必须能在真实环境暂时不可用时继续讲完整故事。

## 完成标准

比赛版本完成时，四名成员都应能回答：

- Dashboard 解决了什么问题；
- 学生如何提交和查看作业；
- Dashboard 如何与 Slurm 交互；
- 当前原型有哪些限制；
- 如果继续开发，下一步是什么。
