# 四人团队协作与一个月排期

## 目标

本项目以比赛 MVP 为目标。四人围绕一个可运行版本协作，优先打通以下闭环：

```text
填写作业参数
  -> 提交 Slurm 作业
  -> 查看排队和运行状态
  -> 查看 stdout/stderr 日志
  -> 取消或克隆作业
  -> 查看基础资源统计
```

每周结束都要有一个可以打开、操作和验证的版本。不要等到最后一周才做首次联调。

## 四人主责

姓名暂以 A、B、C、D 代称，确定成员后替换。主责不等于唯一负责人，每个人都要参与需求、联调、测试和演示。

| 成员 | 主责 | 主要交付 | 固定协作人 |
| --- | --- | --- | --- |
| A | 产品与前端 | 页面原型、作业列表、提交表单、详情、日志和统计视图 | D 负责体验验收，B 负责接口联调 |
| B | 后端与数据 | FastAPI、数据模型、API、业务流程、SQLite 和权限校验 | A 负责接口联调，C 负责 Slurm 边界 |
| C | Slurm 与平台 | `sbatch`、`squeue`、`sacct`、`scancel` 适配器、解析器、平台验证 | B 负责业务集成，D 负责测试记录 |
| D | 测试、部署与材料 | 测试、启动部署、演示数据、README、答辩材料和发布验收 | C 负责平台协作，A 负责演示体验 |

交叉评审安排：

- A 的页面与交互由 D 验收；
- B 的 API 与数据由 A 联调；
- C 的 Slurm 适配器由 B 评审；
- D 的测试与部署由 C 验证。

## 四周排期

### 第 1 周：需求冻结与最小骨架

目标：明确产品范围，完成前后端初始化，页面可以调用后端。

主要工作：

- 全员阅读比赛材料和项目文档，确认用户故事、MVP 范围和不做清单；
- D 建立 Gitee Issues，确定分支规范、评审流程和阶段同步时间；
- B 确定 Job、状态、资源、日志等数据结构，整理 API 草案；
- A 绘制作业列表、提交表单、详情和日志页面流程；
- C 提供 Slurm 输出样例，明确 Job ID、状态、资源和退出码字段；
- B 创建 FastAPI、配置、健康检查和基础数据接口；
- A 创建 React 页面骨架和基础布局；
- D 准备排队、运行、成功、失败四类 Mock 数据和测试目录；
- 全员完成第一次前后端联调。

本周交付：

- 产品流程图；
- API 和数据模型草案；
- 前后端可启动项目；
- Mock 作业列表和基本详情；
- Gitee Issue 任务表。

本周验收：浏览器可以打开页面，前端可以调用后端，用户可以看到 Mock 作业列表和基本详情。

### 第 2 周：作业只读视图与 Slurm 接入

目标：能够查看当前作业、历史作业和作业详情，Mock 与真实数据字段一致。

主要工作：

- C 设计 `squeue`、`sacct` 的参数数组、超时、错误处理和结构化解析；
- C 建立脱敏 Slurm fixtures，完成 Job ID、状态、节点、退出码和资源字段解析；
- B 接入当前作业和历史作业查询，支持筛选、排序、分页和适配器切换；
- A 完成作业列表和详情页，展示状态、资源、节点、时间、退出码和排队原因；
- B 明确用户与 Job ID 的映射，完成详情查询的所有权校验；
- D 编写状态映射、接口错误和越权访问测试；
- C 在平台进行最小资源、最短时长的真实作业验证，并记录证据；
- 全员完成真实数据与 Mock 数据的字段核对。

本周交付：

- Slurm 适配器接口；
- 状态解析器和脱敏 fixtures；
- 当前/历史作业 API；
- 作业列表和详情页；
- 状态、错误和越权测试；
- 平台验证记录。

本周验收：学生可以查看自己的当前作业和历史作业；Slurm 不可用时页面能给出明确提示；越权查询会被拒绝。

### 第 3 周：提交、日志与取消

目标：完成 Dashboard 的核心操作链路，并通过最小真实作业验证。

主要工作：

- B、C 确定分区、QoS、CPU、内存、GPU、时长和命令边界；
- C 增加资源上限校验，生成受控作业脚本，使用参数数组调用 `sbatch`；
- A 完成提交表单，增加必填、范围、格式、提交中和错误状态；
- B 完成提交 API，保存用户、参数、Job ID 和时间；
- B、C 处理重复点击、提交超时、Slurm 失败和数据库异常一致性；
- C、B 完成 stdout/stderr 读取，处理文件不存在、权限不足、作业未开始和增量刷新；
- B、C 完成取消 API 和 `scancel` 调用，取消前校验归属与状态；
- A 完成日志展示、取消确认和操作反馈；
- D 编写提交、日志、取消、非法参数和重复操作测试；
- 全员联调完整流程。

本周交付：

- 结构化提交表单和参数校验；
- 受控 `sbatch` 提交流程；
- 提交 API 和数据库记录；
- stdout/stderr 日志查看；
- 作业取消功能；
- 核心流程测试记录。

本周验收：合法作业可以进入队列，非法参数在 Web 层拒绝，用户只能读取和取消自己的作业。

### 第 4 周：克隆、统计与比赛版本

目标：补齐展示功能，完成可部署、可恢复、可讲解的版本。

主要工作：

- B 完成作业克隆：读取旧参数但重新校验、重新提交，不复用旧 Job ID；
- C、B 解析 CPU、GPU、内存和运行时长，区分申请资源与实际使用资源；
- A 完成克隆入口、资源统计视图和页面统一；
- D 完成接口、权限、提交、日志、取消、克隆、统计和异常场景测试；
- D 在目标环境演练启动、SQLite 初始化、服务重启和 Job ID 状态恢复；
- 全员审查命令拼接、敏感信息、运行时文件、文档一致性和主分支可运行性；
- A、D 完成架构图、产品介绍、演示脚本、答辩问题和限制说明；
- 全员准备真实平台、预提交作业和 Mock 三套演示路径。

本周交付：

- 作业克隆功能；
- CPU、GPU、内存和时长统计；
- 完整测试报告；
- 目标环境部署记录；
- 演示脚本、架构图和答辩材料；
- 可发布版本候选。

本周验收：功能在目标环境可启动，核心流程可重复演示，服务重启后能恢复必要的作业状态。

## 最后两天：冻结与模拟比赛

### 倒数第 2 天：三路演示

全员各完成一次完整演示，并准备三套路径：

- A 路径：连接真实平台并提交作业；
- B 路径：使用预先提交的真实作业和日志；
- C 路径：Slurm 不可用时使用脱敏 Mock 数据。

每个人都要能回答：项目解决什么问题、Dashboard 如何调用 Slurm、如何限制资源、如何校验作业归属、当前版本有哪些限制。

### 最后 1 天：只修阻断问题

只处理以下问题：无法启动、无法提交、状态错误、日志无法查看、取消或克隆失败、权限越界、页面严重错位和演示数据缺失。

当天不新增大功能、不做架构重构、不随意升级依赖。完成代码审查、打发布 tag，并保存演示版本、Mock 数据和演示视频。

## 每周协作机制

每周固定进行一次阶段同步，每人回答五件事：

1. 本周完成了什么；
2. 下周准备完成什么；
3. 当前阻塞是什么；
4. 是否影响其他人的接口；
5. 下周结束时能否演示。

每周结束更新：

- Gitee Issue 状态；
- 当前可运行版本；
- 已知问题、优先级和负责人；
- 下一周的验收标准。

超过半小时的工作必须建立 Issue，写清目标、用户可见结果、范围、验收标准、负责人和截止时间。

## 分支、评审与安全

`master` 始终保持可演示。功能使用短期分支，例如：

```text
feature/job-submit
feature/slurm-adapter
fix/log-tail
docs/demo-script
```

功能代码通过 Pull Request 合并，至少一名队友评审。提交信息使用 `feat:`、`fix:`、`docs:`、`test:` 或 `chore:` 前缀。

### 功能分支分配

最小骨架完成后按以下方式拆分任务。当前只有项目管理员参与开发时，可以暂时直接在 `master` 上完成初始化和第一版业务闭环；开始多人并行开发后，必须切换到功能分支和 PR 流程。分支只承担一个功能边界；如果一个功能需要多人协作，仍由一人负责分支和 PR，其他人通过评审或联调参与。

| 负责人 | 必须分支 | 主要职责 |
| --- | --- | --- |
| A（前端） | `feature/frontend-shell`、`feature/job-list`、`feature/job-submit`、`feature/job-logs` | 页面、表单、列表、详情、日志和操作反馈 |
| B（后端） | `feature/backend-health`、`feature/job-cancel`、`feature/job-clone` | API、业务服务、数据模型、权限和操作流程 |
| C（Slurm） | `feature/slurm-adapter`、`feature/job-usage` | 命令执行、输出解析、资源统计和平台验证 |
| D（测试部署） | `feature/mvp-integration` | fixtures、测试、部署演练、Mock 演示和回归验收 |

创新功能在核心功能合并后再排期，优先顺序建议为：`feature/demo-mode`、
`feature/live-job-refresh`、`feature/resource-charts`、`feature/job-templates`，
其他创新项视比赛剩余时间决定。创新功能不得回头破坏核心分支的验收标准。

每个 PR 至少写明：对应计划项、依赖分支、用户可见结果、测试命令和未解决风险。

### 并行开发与冲突控制流程

多人同时开发时，采用“目录隔离、接口先行、小步合并、及时同步”的方式协作。

#### 1. 开工前确认任务和边界

开发者先从最新 `master` 创建分支，并在 Issue 或群内声明分支、负责人、修改目录和依赖关系：

```bash
git switch master
git pull --ff-only
git switch -c feature/job-list
```

推荐的目录责任边界如下：

| 负责人 | 优先负责目录 | 协作说明 |
| --- | --- | --- |
| A | `frontend/src/features/jobs/`、页面和样式 | 通过约定好的 API 与 B 联调 |
| B | `backend/app/api/`、`services/`、`schemas/`、`repositories/` | 先发布接口字段和错误格式 |
| C | `backend/app/slurm/`、`fixtures/slurm/` | 只负责受控命令、解析和平台证据 |
| D | `backend/tests/`、`tests/e2e/`、`deploy/` | 负责回归、部署和演示验收 |

尽量不要多人同时修改 `README.md`、`package.json`、`backend/app/main.py`、全局样式和同一份文档。确需修改共享文件时，应提前通知相关负责人。

#### 2. 接口先行，前后端并行

后端和前端开发前先确定请求路径、字段、状态码和错误响应。例如作业功能至少约定：

```text
GET  /api/jobs
GET  /api/jobs/{job_id}
POST /api/jobs
POST /api/jobs/{job_id}/cancel
POST /api/jobs/{job_id}/clone
GET  /api/jobs/{job_id}/logs
```

后端负责实现真实接口，前端可以先使用与接口字段一致的 Mock 数据开发。接口变更必须同步更新 schema、前端类型、测试和 PR 描述。

#### 3. 小步提交并及时同步

一个分支只解决一个功能，完成一个可验证的小单元就提交。功能分支开发期间定期同步 `master`：

```bash
git fetch origin
git rebase origin/master
```

如果分支已经被多人共同使用，不要擅自 rebase；改用合并 `origin/master`，或先和协作者确认。合并前必须解决冲突并重新运行相关测试。

#### 4. PR 评审和合并

PR 的目标分支固定为 `master`。PR 描述至少包含：对应计划项、实现内容、依赖分支、测试命令、测试结果和已知风险。至少一名队友评审通过后才能合并。

评审重点包括功能验收、接口兼容性、权限校验、Slurm 安全边界、测试覆盖和运行时文件检查。基础功能建议使用 Squash Merge，保持 `master` 提交历史清晰。

#### 5. 冲突处理

发现冲突时先保存当前工作，再同步目标分支，不要直接覆盖双方修改：

```bash
git status
git fetch origin
git rebase origin/master
git status
```

逐个检查冲突文件，确认保留内容后执行：

```bash
git add <resolved-file>
git rebase --continue
git push --force-with-lease
```

如果无法判断某段代码的归属，应暂停合并并邀请原作者共同确认。禁止使用 `git reset --hard` 或批量丢弃修改解决冲突。

#### 6. 合并后的回归

PR 合并后，所有开发者更新本地 `master`，再从最新版本创建后续分支。合并人需要在 PR 中记录最终测试结果；如果影响公共接口，应同步更新计划、README 或架构文档。

Slurm 相关工作必须遵守以下边界：

- 使用参数数组和结构化解析，禁止拼接未经校验的 shell 命令；
- 学生计算任务只能通过 Slurm 提交，不能在登录节点直接运行；
- 读取日志、取消、克隆和查看详情前必须校验作业所有权；
- 优先使用脱敏 fixtures，真实平台验证只使用最小资源和最短时长；
- 不提交密码、Token、SSH 私钥、真实用户日志、SQLite 文件或构建产物。

## 完成标准

比赛版本合并前，必须满足：

- 四人都能启动项目并完成核心演示；
- 四人都理解前后端、Slurm、权限和部署边界；
- 关键操作有测试或手工验收记录；
- 有真实平台、预提交作业和 Mock 三套演示路径；
- README、架构、环境、协作和部署文档保持一致；
- `master` 可以从干净环境启动，且没有敏感信息或运行时产物。

## 协作工具

| 工具 | 用途 |
| --- | --- |
| Gitee 仓库 | 代码、文档和版本历史 |
| Gitee Issues | 任务、Bug、验收标准和负责人 |
| Gitee Pull Requests | 评审、验证记录和合并 |
| QQ 或微信群 | 快速沟通、紧急问题和会议通知 |
