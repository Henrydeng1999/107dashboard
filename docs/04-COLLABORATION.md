# 四人团队协作与一个月排期

## 目标

本项目以比赛 MVP 为目标。四人围绕一个可运行版本协作，优先打通以下闭环：

```text
填写作业参数
  -> 提交 Slurm 作业
  -> 查看排队和运行状态
  -> 查看 stdout/stderr 日志
  -> 取消或克隆作业
  -> 查看基础资源统计
```

每天结束都要有一个可以打开、操作或验证的版本。不要等到最后一周才做首次联调。

## 四人主责

姓名暂以 A、B、C、D 代称，确定成员后替换。主责不等于唯一负责人，每个人都要参与需求、联调、测试和演示。

| 成员 | 主责 | 主要交付 | 固定协作人 |
| --- | --- | --- | --- |
| A | 产品与前端 | 页面原型、作业列表、提交表单、详情、日志和统计视图 | D 负责体验验收，B 负责接口联调 |
| B | 后端与数据 | FastAPI、数据模型、API、业务流程、SQLite 和权限校验 | A 负责接口联调，C 负责 Slurm 边界 |
| C | Slurm 与平台 | `sbatch`、`squeue`、`sacct`、`scancel` 适配器、解析器、平台验证 | B 负责业务集成，D 负责测试记录 |
| D | 测试、部署与材料 | 测试、启动部署、演示数据、README、答辩材料和发布验收 | C 负责平台协作，A 负责演示体验 |

交叉评审安排：

- A 的页面与交互由 D 验收；
- B 的 API 与数据由 A 联调；
- C 的 Slurm 适配器由 B 评审；
- D 的测试与部署由 C 验证。

## 30 天排期

### 第 1 周：需求冻结与最小骨架

目标：项目能启动，前后端能通信，团队对范围和接口有共同认识。

| 天数 | 主责 | 协作 | 工作与交付 |
| --- | --- | --- | --- |
| 第 1 天 | D | 全员 | 阅读比赛材料和项目文档；确认用户故事、MVP 范围和不做清单；建立 Gitee Issues；确定分支和每日同步时间 |
| 第 2 天 | B | A、C、D | 确定 Job、状态、资源和日志数据结构；整理 API 草案；A 绘制页面流程；C 提供 Slurm 输出样例 |
| 第 3 天 | B、A | C、D | 创建 FastAPI、React、配置和健康检查接口；完成基础布局；整理启动说明 |
| 第 4 天 | A | B、D、C | 完成 Mock 作业列表；准备排队、运行、成功、失败样例；确认页面字段含义 |
| 第 5 天 | B | A、C、D | 完成作业列表 API 联调；统一加载、空数据和错误状态；增加第一批接口测试 |
| 第 6 天 | D | 全员 | 验收前后端启动、健康检查、Mock 列表和作业详情；记录阻塞问题和负责人 |
| 第 7 天 | 全员 | D 记录 | 周复盘；冻结第二周接口；确认平台待确认事项和下周可演示目标 |

第一周验收：浏览器可以打开页面，前端可以调用后端，用户可以看到 Mock 作业列表和基本详情。

### 第 2 周：作业只读视图与 Slurm 接入

目标：能够查看当前作业、历史作业和作业详情，Mock 与真实数据字段一致。

| 天数 | 主责 | 协作 | 工作与交付 |
| --- | --- | --- | --- |
| 第 8 天 | C | B、D | 设计 Slurm 命令参数数组、超时和错误处理；建立脱敏 fixtures |
| 第 9 天 | C | B、A | 完成状态、Job ID、节点、退出码和资源字段解析；处理未知状态 |
| 第 10 天 | B | A、C、D | 接入当前和历史作业查询；支持筛选、排序、分页和 Mock/真实适配器切换 |
| 第 11 天 | A | B、C | 完成详情页；展示状态、资源、节点、时间、退出码和排队原因 |
| 第 12 天 | B | C、D | 明确用户与 Job ID 映射；完成详情查询的所有权校验和越权测试 |
| 第 13 天 | C | B、D | 在平台做最小真实作业验证；记录 Job ID、资源、状态、退出码和证据 |
| 第 14 天 | D | 全员 | 验收当前/历史作业、状态展示、错误提示、Mock/真实字段一致性和越权拒绝 |

第二周验收：学生可以查看自己的当前作业和历史作业，Slurm 不可用时页面仍能给出明确提示。

### 第 3 周：提交、日志与取消

目标：完成核心操作链路，并通过最小真实作业验证。

| 天数 | 主责 | 协作 | 工作与交付 |
| --- | --- | --- | --- |
| 第 15 天 | B、C | A、D | 确定分区、QoS、CPU、内存、GPU、时长和命令边界；增加资源上限 |
| 第 16 天 | A | B、C | 完成提交表单；增加必填、范围、格式、提交中和错误状态 |
| 第 17 天 | C | B、D | 生成受控作业脚本；使用参数数组调用 `sbatch`；解析 Job ID；处理提交失败 |
| 第 18 天 | B | A、C、D | 完成提交 API；保存用户、参数、Job ID 和时间；处理重复点击和异常一致性 |
| 第 19 天 | C、B | A、D | 完成 stdout/stderr 读取；处理文件不存在、权限不足、作业未开始和增量刷新 |
| 第 20 天 | B、C | A、D | 完成取消 API 和 `scancel` 调用；取消前校验归属与状态；增加确认交互 |
| 第 21 天 | 全员 | D 记录 | 联调完整流程：填写、提交、查 Job ID、看状态、看日志、取消和刷新 |

第三周验收：合法作业可以进入队列，非法参数在 Web 层拒绝，用户只能读取和取消自己的作业。

### 第 4 周：克隆、统计与比赛版本

目标：补齐展示功能，完成可部署、可恢复、可讲解的版本。

| 天数 | 主责 | 协作 | 工作与交付 |
| --- | --- | --- | --- |
| 第 22 天 | B | A、C | 完成作业克隆；读取旧参数但重新校验、重新提交，不复用旧 Job ID |
| 第 23 天 | C、B | A、D | 解析 CPU、GPU、内存和运行时长；区分申请资源与实际使用资源 |
| 第 24 天 | A | 全员 | 统一状态颜色、操作入口、错误提示和页面跳转；检查投影和窄屏显示 |
| 第 25 天 | D | 全员 | 完成接口、权限、提交、日志、取消、克隆、统计和异常场景测试 |
| 第 26 天 | D | B、C | 在目标环境演练启动、SQLite 初始化、服务重启和 Job ID 状态恢复 |
| 第 27 天 | D | 全员 | 审查命令拼接、敏感信息、运行时文件、文档一致性和主分支可运行性 |
| 第 28 天 | A、D | B、C | 完成架构图、产品介绍、演示脚本、答辩问题和限制说明 |

第四周验收：功能在目标环境可启动，核心流程可重复演示，服务重启后能恢复必要的作业状态。

## 最后两天：冻结与模拟比赛

### 第 29 天：三路演示

全员各完成一次完整演示，并准备三套路径：

- A 路径：连接真实平台并提交作业；
- B 路径：使用预先提交的真实作业和日志；
- C 路径：Slurm 不可用时使用脱敏 Mock 数据。

每个人都要能回答：项目解决什么问题、Dashboard 如何调用 Slurm、如何限制资源、如何校验作业归属、当前版本有哪些限制。

### 第 30 天：只修阻断问题

只处理以下问题：无法启动、无法提交、状态错误、日志无法查看、取消或克隆失败、权限越界、页面严重错位和演示数据缺失。

当天不新增大功能、不做架构重构、不随意升级依赖。完成代码审查、打发布 tag，并保存演示版本、Mock 数据和演示视频。

## 每日协作机制

每天固定进行一次 10 分钟同步，每人回答五件事：

1. 昨天完成了什么；
2. 今天准备完成什么；
3. 当前阻塞是什么；
4. 是否影响其他人的接口；
5. 今天结束时能否演示。

每天结束更新：

- Gitee Issue 状态；
- 当前可运行版本；
- 已知问题、优先级和负责人。

超过半小时的工作必须建立 Issue，写清目标、用户可见结果、范围、验收标准、负责人和截止时间。

## 分支、评审与安全

`master` 始终保持可演示。功能使用短期分支，例如：

```text
feature/job-submit
feature/slurm-adapter
fix/log-tail
docs/demo-script
```

功能代码通过 Pull Request 合并，至少一名队友评审。提交信息使用 `feat:`、`fix:`、`docs:`、`test:` 或 `chore:` 前缀。

Slurm 相关工作必须遵守以下边界：

- 使用参数数组和结构化解析，禁止拼接未经校验的 shell 命令；
- 学生计算任务只能通过 Slurm 提交，不能在登录节点直接运行；
- 读取日志、取消、克隆和查看详情前必须校验作业所有权；
- 优先使用脱敏 fixtures，真实平台验证只使用最小资源和最短时长；
- 不提交密码、Token、SSH 私钥、真实用户日志、SQLite 文件或构建产物。

## 完成标准

比赛版本合并前，必须满足：

- 四人都能启动项目并完成核心演示；
- 四人都理解前后端、Slurm、权限和部署边界；
- 关键操作有测试或手工验收记录；
- 有真实平台、预提交作业和 Mock 三套演示路径；
- README、架构、环境、协作和部署文档保持一致；
- `master` 可以从干净环境启动，且没有敏感信息或运行时产物。
