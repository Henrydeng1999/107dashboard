# 01 开发计划

本计划用于统一项目初期方向。具体实现可以根据算力平台实际接口和管理员要求调整。

当前项目以比赛 MVP 为目标：先完成可运行、可演示、可讲清楚价值的闭环，生产级部署问题不作为比赛阶段阻塞项。

比赛开发过程中使用 AI 辅助需求整理、代码编写、文档维护和测试，但 AI 不属于 Dashboard 产品功能。

## 功能分支与优先级

最小可用骨架合并到 `master` 并通过启动、构建和基础测试后，再从最新
`master` 创建功能分支。当前仅由项目管理员开发的初始化阶段允许直接在
`master` 推进；多人并行开始后必须切换到功能分支和 PR。`master` 始终保持可启动、可演示。

### 比赛必须完成的功能

| 分支 | 功能 | 依赖 | 最低验收标准 |
| --- | --- | --- | --- |
| `feature/backend-health` | FastAPI 应用、配置、健康检查 | 无 | API 可启动并返回健康状态 |
| `feature/frontend-shell` | React/Vite 页面骨架、布局和 API 客户端 | 无 | 页面可构建并访问健康检查 |
| `feature/slurm-adapter` | `sbatch`、`squeue`、`sacct`、`scancel` 受控调用与解析 | 后端骨架 | 参数数组调用、超时、错误和 fixture 测试齐全 |
| `feature/job-list` | 当前/历史作业列表与详情 | Slurm 适配器、数据模型 | 可展示 Job ID、状态、资源、节点和退出码 |
| `feature/job-submit` | 结构化提交表单和受控 `sbatch` 提交 | Slurm 适配器、作业模型 | 合法请求入队，非法资源请求被拒绝 |
| `feature/job-logs` | stdout/stderr 读取与增量刷新 | 作业详情、所有权校验 | 用户只能查看自己的日志，未开始/缺失日志有明确提示 |
| `feature/job-cancel` | 取消作业 | 作业详情、所有权校验 | 只能取消自己的可取消作业，并显示结果 |
| `feature/job-clone` | 从历史作业复制参数并重新提交 | 提交、详情 | 重新校验参数且生成新的 Slurm Job ID |
| `feature/job-usage` | CPU、GPU、内存和运行时长统计 | `sacct`、作业详情 | 区分申请资源与实际使用资源，缺失数据可解释 |
| `feature/mvp-integration` | 前后端联调、部署、演示和回归测试 | 上述核心分支 | 真实平台、预提交作业和 Mock 三条路径均可演示 |

### 可选创新功能

以下功能不应阻塞核心闭环，建议在对应必须功能合并后再创建：

| 分支 | 创新方向 | 建议前置条件 | 创新价值 |
| --- | --- | --- | --- |
| `feature/job-templates` | 常用作业模板、课程模板和一键复用 | 提交、克隆 | 降低重复配置成本 |
| `feature/live-job-refresh` | 轮询策略、状态变化提示和页面自动刷新 | 作业列表 | 提升运行过程可见性 |
| `feature/resource-charts` | 资源使用趋势图和用户统计卡片 | 资源统计 | 更直观展示算力使用情况 |
| `feature/job-search` | 按状态、时间、名称和分区筛选 | 作业列表 | 提高历史作业定位效率 |
| `feature/failure-hints` | 根据退出码和日志特征给出排查提示 | 日志、统计 | 帮助学生定位常见失败原因 |
| `feature/demo-mode` | Slurm 不可用时自动切换脱敏 Mock 数据 | 核心页面、fixtures | 提高比赛现场演示稳定性 |
| `feature/accessibility-polish` | 键盘操作、移动端布局和无障碍提示 | 前端骨架 | 提升产品完成度 |

创新功能必须满足：不绕过后端权限校验、不直接执行用户 shell 字符串、不开启额外生产基础设施，并且不能破坏 Mock 和真实 Slurm 两种数据路径。

## 阶段 3：作业提交与控制

- [x] 设计结构化作业提交参数
- [x] 生成受控 `sbatch` 脚本
- [x] 增加资源上限、运行时长上限和并发限制
- [x] 支持取消作业
- [x] 支持基于历史作业克隆并重新提交

验收标准：合法提交可以进入 Slurm 队列，非法或超限参数在 Web 层被拒绝。

Native 提交链已具备 `python`/`python3` 窄命令白名单、固定平台参数与资源上限、服务端作业目录、参数数组 `sbatch --parsable`、Job ID 严格解析、回执、owner 元数据和脱敏审计。107 无写入预检和唯一一次最小真实作业均已通过：Job `24011` 正常完成并返回 `0:0`。Native HTTP 路由增加持久化幂等键和默认 1 个活跃作业的并发门禁，并已在 107 通过不调用 `sbatch` 的 `400/422` 门禁检查；部署开关继续默认关闭，长期开放需单独决定。

Native 取消和克隆已接入独立默认关闭门禁。取消在 `scancel` 前依次要求有效 UID、owner 可见状态、同 owner 的 Native 元数据和持久化幂等占位，并记录脱敏审计；克隆从可信元数据恢复原参数，再经过 schema、命令白名单、资源上限、活跃作业上限和 namespaced idempotency 后调用 `sbatch`。真实平台验收使用脚本自建的两个最小作业并全部取消，不操作外部作业。

## 阶段 4：日志和资源可视化

- [x] 使用脱敏 Fixture 展示 stdout/stderr 日志
- [x] 支持基于字节偏移的日志增量刷新
- [x] 展示申请/分配的 GPU、CPU、内存及实际运行时长
- [x] 增加作业级和当前用户基础统计

验收标准：学生可以定位作业失败原因，并查看作业实际资源使用情况。

Native stdout/stderr 已完成本地受控实现：API 不接受路径，只按可信 owner 和 Slurm Job ID 查询持久化元数据，并限制为 `JOB_WORKSPACE_DIRECTORY/submission-<id>/stdout.log|stderr.log`。增量偏移、缺失文件、跨 owner、目录穿越、非普通文件和脱敏错误均有回归；提交 `beb39f7` 已在 107 对 Job `24011` 通过不打开文件的路径预检。`NATIVE_LOGS_ENABLED` 默认关闭，真实日志只在一次性验收 shell 中临时开启。

2026-07-18 的集中验收已完成真实日志限量读取和 Native 控制闭环：Job `24011` 的 stdout 读取 14 字节、stderr 读取 0 字节且正文未回显；脚本自建 Job `24063`、`24064` 后分别完成取消，`squeue` 无遗留活动作业，幂等记录和审计链完整。前端演示体验已加入自适应刷新、状态变化提示、资源对比、常用模板和保守排障提示。当前批次进一步增加 Native 读取失败后的显式只读 Fixture 回退、动态能力声明、静态前端托管、用户级服务模板和无写入发布验收脚本；平台集中回归通过后进入最终演示排练。

提交 `b253ac0` 的发布验收已在 107 通过：Native 列表与摘要均返回 4 个真实作业，模拟故障展示 5 个脱敏 Fixture 作业，写请求稳定返回 503 且未调用 Slurm 写命令。发布安全门已经完成，剩余工作集中在预构建前端传输、服务启动和完整页面演示回归。

2026-07-19 开始 Native 全交互批次。最低闭环不是分别调用孤立脚本，而是经同一 FastAPI HTTP 应用依次完成：提交一个可成功结束的最小作业、等待 `COMPLETED/0:0`、限量读取 stdout/stderr、读取 usage；再提交一个可取消作业、取消、从可信元数据克隆并取消克隆。集中脚本固定为 3 个最小 CPU 作业、2 次取消，失败时只清理本脚本创建且仍活动的作业，最终核对列表/摘要、5 条幂等记录与 `SBATCH_ACCEPTED`/`SCANCEL_ACCEPTED` 审计链。平台验收通过后，才允许采用 `deploy/107-native-interactive.env.example` 作为真实交互部署基线。

提交 `85c9646` 已在 107 完成本批次验收。Job `24159` 为 `COMPLETED/0:0`，stdout/stderr 分别限量读取 14/0 字节，usage 返回申请、分配、运行时长、峰值内存和 CPU 时间；Job `24160` 与克隆 Job `24161` 均由 UID `68311` 取消。五条幂等记录全部成功，审计链完整，`squeue` 无活动作业。下一批转入真实 Web 服务和浏览器端到端联调，再扩展更贴近学生任务的脚本与环境配置能力。

基础产品服务批次要求正常运行只展示真实 Native 数据，禁用 Fixture 回退；用户目录提供配置、启动、停止、重启、状态与日志入口，并拒绝旧前端构建。前端默认模板必须在当前窄命令策略和受控工作目录下真实可运行。完成标准是在浏览器依次完成“提交并观察完成作业、查看日志和 usage、提交并取消活动作业、克隆并取消克隆”，且页面和服务检查均证明 Fixture 不参与结果。

## 待确认问题

- 学生如何登录：校园统一认证、平台现有账号，还是临时账号？
- 平台是否允许为生产服务账号启用 linger，或提供系统级 service？
- 平台如何提供反向代理、HTTPS 和访问域名？
- Web 服务运行在哪个节点？是否允许访问计算节点日志目录？
- 作业代码如何进入平台：上传、Git 仓库、共享目录，还是多种方式并存？
- 学生是否允许自定义 shell 命令？如果允许，限制边界是什么？
- 日志保留多久，作业结果和文件由谁负责清理？
- 不同课程或用户组是否有不同的分区和资源配额？
