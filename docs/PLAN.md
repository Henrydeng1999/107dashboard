# 开发计划

本计划用于统一项目初期方向。具体实现可以根据算力平台实际接口和管理员要求调整。

## 阶段 0：项目初始化

- [x] 建立项目 README、架构说明和开发计划
- [ ] 确定前端、后端和 Python/Node 版本
- [ ] 确定 Docker Compose 的服务划分
- [ ] 确定开发、测试、生产三套配置边界
- [ ] 确定学生身份认证方式

验收标准：新成员可以仅通过 README 了解项目目标、技术栈、部署方向和后续任务。

## 阶段 1：最小可用骨架

- [ ] 创建 FastAPI 后端和健康检查接口
- [ ] 创建 React 前端和基础布局
- [ ] 增加 Dockerfile、Compose 文件和环境变量模板
- [ ] 增加统一日志、错误响应和基础配置加载
- [ ] 增加 CI 中的格式检查和基础测试

验收标准：Docker Compose 启动后，浏览器可以打开前端，前端可以调用后端健康检查接口。

## Slurm 集成边界

- [ ] 将现有 `gpu-*` 命令作为开发和运维参考，不直接作为 Dashboard 产品依赖
- [ ] 确定 API 服务调用 Slurm 的方式：容器内客户端或受控宿主机代理
- [ ] 设计用户、作业、Slurm Job ID、资源申请和日志路径的映射
- [ ] 明确 SSH 私钥、ControlMaster socket 和 tmux 不进入产品运行时容器
- [ ] 增加重复请求、Slurm 不可用和作业状态延迟的处理

验收标准：Dashboard 可以通过受控 Slurm 集成提交和查询作业；产品核心流程不依赖开发机上的 SSH 控制连接或 tmux 会话。

## 阶段 2：作业只读视图

- [ ] 接入 `squeue`，展示排队和运行中的作业
- [ ] 接入 `sacct`，展示历史作业和退出状态
- [ ] 展示作业详情、节点、资源申请和时间信息
- [ ] 增加作业筛选、分页和错误状态

验收标准：学生可以在网页中查看自己的当前作业和历史作业，不影响现有 Slurm 作业。

## 阶段 3：作业提交与控制

- [ ] 设计结构化作业提交参数
- [ ] 生成受控 `sbatch` 脚本
- [ ] 增加资源上限、运行时长上限和并发限制
- [ ] 支持取消作业
- [ ] 支持基于历史作业克隆并重新提交

验收标准：合法提交可以进入 Slurm 队列，非法或超限参数在 Web 层被拒绝。

## 阶段 4：日志和资源可视化

- [ ] 展示 stdout/stderr 日志
- [ ] 支持日志增量刷新
- [ ] 展示 GPU、CPU、内存和运行时长
- [ ] 增加作业级和用户级基础统计

验收标准：学生可以定位作业失败原因，并查看作业实际资源使用情况。

## 阶段 5：生产化部署

- [ ] 完成反向代理和 HTTPS 配置
- [ ] 配置数据库备份和日志轮转
- [ ] 增加服务健康检查和重启策略
- [ ] 增加管理员操作审计
- [ ] 完成并发、权限和异常场景测试

验收标准：服务可以在服务器 Docker 环境中稳定运行，服务重启不会丢失必要的作业元数据。

## 待确认问题

- 学生如何登录：校园统一认证、平台现有账号，还是临时账号？
- 后端容器是否可以直接调用 Slurm 命令？
- Web 服务运行在哪个节点？是否允许访问计算节点日志目录？
- 作业代码如何进入平台：上传、Git 仓库、共享目录，还是多种方式并存？
- 学生是否允许自定义 shell 命令？如果允许，限制边界是什么？
- 日志保留多久，作业结果和文件由谁负责清理？
- 不同课程或用户组是否有不同的分区和资源配额？
